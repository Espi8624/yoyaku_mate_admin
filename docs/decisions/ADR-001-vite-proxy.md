# ADR-001: Vite開発サーバープロキシの採用

> 作成日: 2026-07-10  
> ステータス: 改定済み (Revised / ビルドモード別の単一バインディングへ改修)

## コンテキスト

Adminダッシュボードから外部バックエンドサーバーへ直接リクエストを送る場合、CORS問題が発生します。
以前はUIの開発(Dev)/本番(Prod)タブを通じて両方の環境に同時アクセスできるように二重プロキシ設定を採用していましたが、これによりローカル開発時にDBとバックエンドが明確に区別されず混線を引き起こしたため、**起動/ビルドモード別の単一バインディング方式**へ改定されました。

---

## 検討対象

| 方法 | 説明 | 課題 |
|------|------|--------|
| バックエンドでCORS許可 | `Access-Control-Allow-Origin: *` | 管理者用APIのエンドポイントが不用意に露出 |
| 独自の Nginx プロキシ | ローカル上に別途プロキシサーバーを構築 | ローカル開発環境のインフラ構築の手間が増加 |
| **Vite 開発サーバープロキシ** | `vite.config.js` の `proxy` 設定 | 開発時のみ有効、ビルド結果には影響せず安全 |

---

## 決定: Vite開発サーバー単一プロキシの採用

### 2026-07-10 改定事項

UI上のタブ切り替えに伴う動的なAPIアドレスの切り替え方式を廃止し、`npm run dev:dev` および `npm run dev:prod` のように起動する実行環境(`.env`)ファイルに合わせて、自動的に単一のバックエンドに連動するようにプロキシを一元化しました。

1. **安全性の確保**: ビルド/起動時に参照するDB/バックエンドターゲットが確定するため、開発中に誤って本番APIを呼び出したり、誤ったDB接続先へリクエストを送るような大きな事故を未然に防ぎます。

2. **Vite単一プロキシの統合**: `/proxy-dev`、`/proxy-prod` の代わりに `/api/admin` 単一のエンドポイントへプロキシ設定を簡素化しました。

3. **CORS回避の維持**: ローカル開発時はViteプロキシサーバーを通じて外部APIを呼び出すことで、引き続きCORS問題を安全に回避します。

### トレードオフ (許容可能)

- ローカル環境でのみ動作するため、Vercel等への本番配備時には対応する直接宛先の環境変数 (`VITE_PROXY_TARGET`) の事前設定が必要。
- Viteツールへの密結合。ビルドツール変更時にはプロキシ設定の再定義が必要。
