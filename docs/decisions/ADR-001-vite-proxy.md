# ADR-001: Vite 開発サーバープロキシの採用

> 作成日: 2026-07-10  
> ステータス: 確定

## コンテキスト

Admin大画面ダッシュボード上で、開発(Dev)環境と本番(Prod)環境のバックエンドサーバーをタブ切り替えによって同時に管理する必要があります。  
ローカル開発環境から外部の各バックエンドAPIサーバーへ直接リクエストする際、CORS問題が発生します。

---

## 検討対象

| 方法 | 説明 | 課題 |
|------|------|--------|
| バックエンドでCORS許可 | `Access-Control-Allow-Origin: *` | 管理者用APIのエンドポイントが不用意に露出 |
| 独自の Nginx プロキシ | ローカル上に別途プロキシサーバーを構築 | ローカル開発環境のインフラ構築の手間が増加 |
| **Vite 開発サーバープロキシ** | `vite.config.js` の `proxy` 設定 | 開発時のみ有効、ビルド結果には影響せず安全 |

---

## 決定: Vite 開発サーバーの二重プロキシ採用

### 理由

1. **バックエンドのコード修正不要**: バックエンド側のCORS設定を変更することなく、ローカル開発環境から直接安全に外部APIを呼び出すことができます。

2. **2つの環境への同時接続性**: `/proxy-dev` と `/proxy-prod` のパスプレフィックスを設けることで、1つのWebダッシュボード上から両サーバー環境へ同時にアクセスできます。

3. **ビルド成果物との分離**: Viteのプロキシ機能はローカルの `npm run dev` 実行時のみ動作するため、本番ビルド（Vercel配備等）後は直接通信を行う形になり、本番環境の無駄なオーバーヘッドがありません。

### トレードオフ (許容可能)

- ローカル環境でのみ動作するため、Vercel等への本番配備時には対応する直接宛先の環境変数 (`VITE_PROXY_DEV_TARGET`, `VITE_PROXY_PROD_TARGET`) の事前設定が必要。
- Viteツールへの密結合。ビルドツール変更時にはプロキシ設定の再定義が必要。
